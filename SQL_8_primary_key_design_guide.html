🔑 主キー設計の判断基準

💭 主キー選択のフローチャート
❓ 業務上の自然なキー（学生ID、商品コードなど）は存在しますか？
✅ YES
→ ナチュラルキーを検討
❌ NO
→ サロゲートキーを使用
❓ そのキーは将来的に変更される可能性がありますか？
❌ 変更の可能性あり
→ サロゲートキーを使用
✅ 変更されない
→ ナチュラルキーを使用
❓ 単一の属性では一意性を保証できませんか？
✅ YES
→ 複合主キーを検討
❌ NO
→ 単一キーを使用
🤖 サロゲートキー
システムが自動生成する人工的なキー。 業務には直接関係しない連番やUUID。
CREATE TABLE student (
id INT AUTO_INCREMENT PRIMARY KEY,
student_id VARCHAR(10) NOT NULL UNIQUE,
name VARCHAR(100) NOT NULL
);
-- 使用例 --
INSERT INTO student (student_id, name)
VALUES ('S001', '田中太郎');
→ id = 1 が自動採番
✅ メリット
• 絶対に変更されない
• 一意性が保証される
• システムが自動管理
• 外部キー参照が安定
❌ デメリット
• 業務的意味がない
• JOIN時に可読性低下
• 追加のインデックス必要
• デバッグ時に分かりにくい
🏷️ ナチュラルキー
業務上で実際に使用される自然なキー。 学生ID、商品コード、ISBN等。
CREATE TABLE subject (
subject_code VARCHAR(10) PRIMARY KEY,
subject_name VARCHAR(100) NOT NULL,
credits INT NOT NULL );
-- 使用例 --
INSERT INTO subject VALUES
('CS101', 'プログラミング基礎', 3),
('MATH201', '微積分I', 4);
✅ メリット
• 業務的意味がある
• JOIN時に分かりやすい
• 検索が直感的
• ストレージ効率が良い
❌ デメリット
• 変更リスクがある
• 一意性の保証が困難
• 文字列型は性能劣化
• 外部システム依存
🔗 複合主キー
複数の属性を組み合わせて一意性を保証。 関連テーブルや履歴テーブルでよく使用。
CREATE TABLE score (
student_id VARCHAR(10),
subject_code VARCHAR(10),
semester VARCHAR(20),
score INT NOT NULL,
PRIMARY KEY (student_id, subject_code, semester)
);
-- 使用例 --
INSERT INTO score VALUES
('S001', 'CS101', '2024春', 85),
('S001', 'CS101', '2024秋', 92);
✅ メリット
• 自然な一意性表現
• 関連データの整合性
• 業務ルールを反映
• 冗長データの防止
❌ デメリット
• 外部キー参照が複雑
• インデックス容量増大
• JOIN性能の劣化
• アプリ開発が複雑
