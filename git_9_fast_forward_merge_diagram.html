🔀 Fast-forward マージの仕組み
📍 マージ前の状態
main:
A
→
B
→
HEAD
feature:
→
B
→
C
→
D
📋 マージ前の状況説明
• mainブランチ：コミットAとBまで進んでいる
• featureブランチ：mainのBから分岐して、CとDのコミットを追加
• mainブランチには、feature分岐後の新しいコミットがない状態
⬇️ git merge feature ⬇️
✅ Fast-forward マージ後の状態
main:
A
→
B
→
C
→
D
← HEAD
🎯 Fast-forward マージの結果
• mainブランチのポインタが D まで「早送り」された
• 新しいマージコミットは作成されない（シンプルで直線的な履歴）
• featureブランチの全ての変更がmainブランチに統合完了
⚙️ Fast-forward マージが可能な条件
1 分岐後の変更
mainブランチに、feature分岐後の新しいコミットがない
2 直線的な履歴
featureブランチがmainの先頭から直線的に伸びている
3 競合なし
同じファイルの同じ部分を変更していない
4 自動実行
条件が揃えばGitが自動的にFast-forwardを選択
✨ Fast-forward マージの利点
🎯 シンプルな履歴: 余計なマージコミットが作られず、履歴が直線的で見やすい
🚀 高速処理: 単純なポインタ移動のため、処理が非常に高速
📖 理解しやすい: 時系列に沿った自然な開発履歴が保たれる
🔄 可逆性: 必要に応じて簡単に元の状態に戻すことが可能
