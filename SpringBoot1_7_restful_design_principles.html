🎯 RESTful設計の基本原則
🎯
リソース指向設計
データやサービスをリソースとして扱い
URLでリソースを一意に識別
/api/students ← 学生コレクション
/api/students/1 ← 個別学生
/api/students/1/courses ← 学生の履修科目
🔄
統一インターフェース
HTTPメソッド（GET/POST/PUT/DELETE）で
操作を統一的に表現
GET → 取得
POST → 作成
PUT → 更新
DELETE → 削除
📊
ステートレス通信
各リクエストは独立して処理
サーバーはクライアント状態を保持しない
// 各リクエストに必要な情報を含める
Authorization: Bearer token
Content-Type: application/json
📝
自己記述的メッセージ
レスポンスにはコンテンツタイプや
処理結果を明確に記述
HTTP/1.1 200 OK
Content-Type: application/json

{"status": "success", "data": {...}}
🌐 リソース指向URL設計パターン
GET
/api/students
全学生の一覧取得
GET
/api/students/{id}
特定学生の詳細取得
POST
/api/students
新しい学生の登録
PUT
/api/students/{id}
学生情報の完全更新
DELETE
/api/students/{id}
学生の削除
GET
/api/students?name=田中&page=1
検索・フィルタリング
✨ RESTful API設計のベストプラクティス
📛
名詞を使用
URLには動詞ではなく名詞を使用
アクションはHTTPメソッドで表現
🔤
複数形を使用
コレクションは複数形で統一
/students, /courses など
🔗
階層構造
関係性をURL階層で表現
/students/1/courses
🎨
一貫性
命名規則とパターンを
API全体で統一
📊
適切なステータス
操作結果に応じたHTTPステータス
コードを返却
📖
文書化
APIの使用方法を明確に文書化
開発者が理解しやすく
⚠️ 避けるべきアンチパターン
動詞を含むURL
❌ /api/getStudents
✅ GET /api/students
不適切なHTTPメソッド
❌ GET /api/deleteStudent/1
✅ DELETE /api/students/1
複雑すぎるURL
❌ /api/students/1/courses/2/grades/3
✅ /api/grades/3
ステータスコード無視
❌ 常に200を返す
✅ 適切なコードを選択
