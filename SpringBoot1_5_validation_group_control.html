従来の課題

🚫 一律のバリデーション

問題のある実装
@Entity
public class Student {
    @NotNull // 常に必須
    private Long studentId;

    @NotBlank
    private String name;
}
問題点:
• 新規登録時: IDは不要（自動生成）なのに必須チェック
• 更新時: IDは必須なのに検証が甘い
• 一つのルールしか設定できない制限
🔄
グループ適用
🎉
グループ制御後

✅ 状況別バリデーション

改善後の実装
@Entity
public class Student {
    @NotNull(groups = UpdateGroup.class)
    private Long studentId; // 更新時のみ必須

    @NotBlank(groups = {CreateGroup.class, UpdateGroup.class})
    private String name; // 常に必須
}
解決効果:
• 新規登録: IDチェックなし、名前は必須
• 更新: ID・名前ともに必須
• 柔軟で適切なバリデーション実現
📝 バリデーショングループの定義
➕
CreateGroup
新規登録時に適用されるバリデーションルール。 IDは不要、必要最小限の項目のみをチェック。
public interface CreateGroup {
    // マーカーインターフェース
    // 実装内容は不要
}
🔄
UpdateGroup
更新時に適用されるバリデーションルール。 IDは必須、既存データの整合性を重視。
public interface UpdateGroup {
    // マーカーインターフェース
    // 実装内容は不要
}
➕
新規登録シナリオ
学生の新規登録では、IDは自動生成されるため不要。 名前やメールアドレスなど基本情報のみを検証。
@NotNull(groups = UpdateGroup.class)
private Long studentId; // 新規時はチェックなし

@NotBlank(groups = {CreateGroup.class, UpdateGroup.class})
private String name; // 新規時も必須

@Email(groups = {CreateGroup.class, UpdateGroup.class})
private String email; // 新規時も必須
適用されるルール:
• studentId: チェックなし（自動生成のため）
• name: 必須チェック
• email: 形式チェック
🔄
更新シナリオ
既存学生の情報更新では、IDは必須（どの学生かを特定するため）。 全ての項目で厳密なバリデーションを実行。
@NotNull(groups = UpdateGroup.class)
private Long studentId; // 更新時は必須

@NotBlank(groups = {CreateGroup.class, UpdateGroup.class})
private String name; // 更新時も必須

@Email(groups = {CreateGroup.class, UpdateGroup.class})
private String email; // 更新時も必須
適用されるルール:
• studentId: 必須チェック（特定のため）
• name: 必須チェック
• email: 形式チェック
• すべての制約が有効
🎮 Controllerでのグループ指定方法
➕ 新規登録メソッド
@PostMapping("/students")
public ResponseEntity<Student> createStudent(
    @Validated(CreateGroup.class) @RequestBody Student student
) {
    // CreateGroupのルールのみ適用
    // studentIdはチェックされない
    return studentService.save(student);
}
🔄 更新メソッド
@PutMapping("/students/{id}")
public ResponseEntity<Student> updateStudent(
    @PathVariable Long id,
    @Validated(UpdateGroup.class) @RequestBody Student student
) {
    // UpdateGroupのルールが適用
    // studentIdが必須チェックされる
    return studentService.update(student);
}
