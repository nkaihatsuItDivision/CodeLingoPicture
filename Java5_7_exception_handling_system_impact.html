例外処理がシステム設計に与える影響

銀行システムでの比較例
❌ 例外を考慮しない設計
危険な銀行口座クラス
public class BankAccount { private double balance; public void withdraw(double amount) { balance -= amount; // 残高不足でも引き落とし？ } public void transfer(BankAccount to, double amount) { this.withdraw(amount); to.deposit(amount); // 途中で失敗したら？ } }
  • データ破損: 残高がマイナスになる
  • 不整合: 送金途中で失敗すると金額が消失
  • デバッグ困難: どこで問題が起きたか不明
  • ユーザー不信: 予期しない動作で信頼失墜
✅ 例外を考慮した設計
堅牢な銀行口座クラス
public class SafeBankAccount { private double balance; public void withdraw(double amount) throws InsufficientFundsException { if (amount > balance) { throw new InsufficientFundsException( "残高不足: " + amount + " > " + balance); } balance -= amount; } public void transfer(SafeBankAccount to, double amount) throws TransferException { try { this.withdraw(amount); to.deposit(amount); } catch (Exception e) { // ロールバック処理 throw new TransferException("送金失敗", e); } } }
  • データ保護: 不正な操作を事前に検知
  • 整合性保証: 途中失敗時のロールバック
  • 問題特定: 明確なエラーメッセージ
  • 信頼性向上: 予測可能で安全な動作
