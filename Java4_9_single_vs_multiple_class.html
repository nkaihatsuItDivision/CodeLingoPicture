単一クラス vs 複数クラス協調の比較
❌
悪い例：単一クラスですべてを管理
Student クラス
フィールド（複雑で管理困難）
private String name;
private String mathSubject;
private int mathScore;
private String mathSemester;
private String englishSubject;
private int englishScore;
private String englishSemester;
// ... 科目が増えるたびに増加
メソッド（責任が混在）
public double calculateMathGrade()
public double calculateEnglishGrade()
public void showMathInfo()
public void showEnglishInfo()
// ... 科目ごとの処理が増加
🚨 問題点
×
科目追加時にクラス全体を修正
×
責任が混在し理解困難
×
コードの重複が多発
×
テストが複雑化
×
バグの影響範囲が広い
✅
良い例：複数クラスで責任分散
Student クラス
学生の責任に集中
private String name;
private List<Grade> grades;
addGrade()
calculateGPA()
showStudentInfo()
Grade クラス
成績の責任に集中
private String subject;
private int score;
private String semester;
getLetterGrade()
isPassing()
getGradePoint()
⟷ クラス間で協調・連携 ⟷
🌟 メリット
✓
責任が明確に分散
✓
新機能追加が容易
✓
保守・修正が簡単
✓
独立してテスト可能
✓
再利用性が高い
📋 設計原則
単一責任の原則：各クラスは一つの責任のみを持つ
疎結合：クラス間の依存関係を最小限に抑える
高凝集：クラス内の要素は密接に関連している
