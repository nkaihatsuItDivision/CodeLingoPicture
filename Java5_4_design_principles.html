ポリモーフィズムと設計原則
開放閉鎖原則（Open-Closed Principle）
「拡張に対して開放的で、変更に対して閉鎖的」なシステム設計
❌ 拡張性の低い設計
public void processStudent(Student student) { if (student.getStudentType().equals("学部生")) { // 学部生固有の処理 } else if (student.getStudentType().equals("大学院生")) { // 大学院生固有の処理 } // 新しい学生種別が追加されるたびに // このメソッドを変更する必要がある }
問題点：
  • 新機能追加時に既存コードを変更
  • 条件分岐の増加でバグリスク増大
  • コードの可読性・保守性が低下
  • テストケースの再実行が必要
✅ 拡張性の高い設計
public void processStudent(Student student) { student.displayInfo(); // 新しい学生種別が student.study(); // 追加されても変更不要 // 各種別のオブジェクトが適切な処理を実行 // ポリモーフィズムにより自動的に // 正しいメソッドが選択される }
利点：
  • 新機能追加時に既存コードを変更不要
  • 条件分岐なしでシンプルな構造
  • 高い可読性・保守性を維持
  • 既存機能のテストは不要
ポリモーフィズムの威力
新しい学生種別（研究生、交換留学生など）を追加しても
既存のprocessStudentメソッドは一切変更不要！

// ✅ 抽象に依存した設計 public class StudentProcessor { // 具体的なクラス（UndergraduateStudent, GraduateStudent）ではなく、 // 抽象的なクラス（Student）に依存 public void processStudents(List<Student> students) { for (Student student : students) { student.displayInfo(); // 抽象メソッド呼び出し } } }
